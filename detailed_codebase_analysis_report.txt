PART 1: THE FRONTEND APPLICATION (CLIENT FOLDER: `/client/components`)
--------------------------------------------------------------------------------

1. FINANCIAL DASHBOARD (Overview)
- File: `overview-dashboard.tsx`
- Functionality: Serves as the executive gateway. 
- Deep Logic: It manages a `fetchOverviewData` method hitting `/orgs/{orgId}/overview`. The state variable `overviewData` contains aggregated arrays of historical burn and future runway. The component calculates "Runway" by finding the total liquid assets (from `Ledger`) divided by trailing 3-month average `OpEx`.
- Embedded Features:
   - `StrategicPulse`: Polling component that highlights organizational health.
   - `Recent Anomalies Alert`: Maps responses from the anomaly detection engine to red/amber warning banners at the top of the interface.

2. FINANCIAL MODELING OVERVIEW & TABS
- File: `financial-modeling.tsx` (2400 lines)
- Functionality: The central brain of the frontend. It manages a persistent polling mechanism `pollModelRunStatus` which pings `/jobs/{jobId}` continuously until the Python-worker transitions the job state to "completed". 
- Tab 1: SaaS/DCF/LBO Output Matrix
   - Contains highly specific conditionally rendered React blocks based on `modelType`. If the user is viewing a "SaaS", Recharts renders LTV:CAC ratios, NRR (Net Retention Rate) > 100% checks, and the Magic Number calculation. If "DCF", it maps terminal flow rates. If "LBO", it renders the `SourcesUses` table.
- Tab 2: Three-Statement Viewer (`statements`)
   - Uses a nested component `<ThreeStatementViewer />` mapping out `summaryJson.statements.incomeStatement`, `balanceSheet`, and `cashFlow`. Every numeric cell has an embedded `onClick` handler that triggers `setProvenanceModalOpen(true)` passing exactly the `cellId`.
- Tab 3: Valuation (`valuation`)
   - Uses the `<FootballFieldChart />` component passing `ranges={valuationSummary}`. Iterates across DCF, transaction multiples, and basic LBO hurdles to render stacked bar charts visualizing Enterprise Value.
   - Houses the `Sensitivity Matrix`: A 2D Table component that loops through `waccSteps` and `growthSteps` outputted by the backend JSON.

3. REAL-TIME SIMULATIONS (The Cockpit)
- File: `realtime-simulations.tsx`
- Functionality: Meeting-ready interactive sandbox. 
- Deep Logic: A massive State construct `SimulationParams` tracks: `monthlyGrowthRate`, `CAC`, `LTV`, `churnRate`, `pricingTier`, `teamSize`, and `marketingSpend`.
- Execution: When a UI slider moves, a debounced API call initiates `POST /orgs/{orgId}/realtime-simulations`. The frontend does NOT calculate complex multi-year depreciation inline; it calls the backend, which passes it to the hyperblock engine. On success, the frontend updates `simulationData`. 
- Autoplay Feature: Pushing the `<Play />` button ticks `currentMonth` up by 1 every 1000ms using a `setInterval` hook, causing the Recharts components to animate sequentially left-to-right.

4. SCENARIO PLANNING 
- File: `scenario-planning.tsx`
- Functionality: Deep version control for finance.
- Deep Logic: Pulls multiple snapshot rows of a base model from Prisma (`Snapshots` relation). It maintains side-by-side array states: `modelBase`, `scenarioA`, `scenarioB`. 
- Execution: The `<ScenarioComparisonView />` maps through `modelBase.metrics` alongside `scenarioA.metrics` and highlights the delta (green for positive EPS impact, red for negative). Utilizes `scenario-snapshot-manager.tsx` to commit user-varied drivers into a deeply frozen database snapshot.

5. AGENTIC CFO / AI ASSISTANT 
- File: `ai-assistant.tsx` & `/ai-assistant` sub-folder
- Functionality: Forensics and autonomous modification tracking.
- Deep Logic: Contains strict typings for `AgentThought`, `AgentVisualization`, and `AgentRecommendation`. 
- Execution: The user inputs a chat prompt. Sends it to `/orgs/{orgId}/ai-cfo/query`.
- The "Staged Changes" Panel: When the Python Reasoning Engine returns `actionable=true`, the recommendation object populates `<StagedChangesPanel />`. This panel features "Approve" / "Reject" triggers. If Approved, the React component fires an explicit `/model/adjustment` PATCH down to the backend modifying the raw baseline parameters. 
- Task Tracker: Any accepted suggestion becomes a persistent `Task[]` rendered in the "Tasks" Tab, storing due dates and integrating out to internal systems.

6. FINANCIAL FORECASTING
- File: `ai-forecasting.tsx`
- Functionality: Time-series statistical output layer.
- Deep Logic: Translates array objects containing `p10`, `p50`, and `p90` intervals returned by the Prophet/ARIMA engine. Maps them onto an `<AreaChart>` from Recharts where the central line is P50 and the bounded colored area is the spread between P10 and P90, representing the interval of confidence. 

7. MONTE CARLO RISK HUB
- File: `monte-carlo-forecasting.tsx`
- Functionality: Visual interface for chaotic stochastic simulation bounds.
- Deep Logic: Passes stochastic bounds (standard deviation limits for growth/churn) out to `/orgs/.../risk` endpoints. Receives an array of `10,000` simulated cash-out dates, turning them into a `<BarChart>` histogram representing the probability density function (PDF) of bankruptcy versus hyper-growth.

8. INVESTOR DASHBOARD
- File: `investor-dashboard.tsx`
- Functionality: Read-only external portal for LPs.
- Deep Logic: Implements a heavily restricted view pulling from `/investor/reports`. Suppresses granular P&L components and surfaces aggregate KPIs, Board Summaries, and the `FootballFieldChart`. 

9. BOARD REPORTING & ANALYTICS
- File: `board-reporting.tsx`
- Functionality: Hard asset generation tool.
- Deep Logic: Triggers PDF and PPTX queue workers. Sends a command to the backend, rendering `<ExportJobQueue />`. A background poller watches for completion and yields a `<a href={downloadUrl}>` element for the finalized presentation.

10. SEMANTIC LEDGER
- File: `semantic-ledger.tsx`
- Functionality: Unified data source truth tracking.
- Deep Logic: Fetches `/transactions/normalized`. Mapped columns: `Source (Stripe/QB/Excel)`, `Raw_Value`, `Standard_Account (CoA)`, `Confidence_Score`. Contains inline sorting and cross-referencing UI.

--------------------------------------------------------------------------------
PART 2: THE BACKEND (EXPRESS / PRISMA -> /backend)
--------------------------------------------------------------------------------

The Backend is the secure traffic cop between the frontend React layer and the Python calculation worker.

1. `model.routes.ts`
- Logic: Takes raw frontend edits, validates against the Prisma schema. Updates `Model` tables, then pushes a job onto the database queue using `insert into Jobs`.
2. `aicfo.routes.ts`
- Logic: Verifies auth tokens. Pulls the `Model_JSON` currently active for the Organization. Compresses it alongside the user's natural language string, pushes to the Python reasoning engine over an internal network call.
3. `provenance.routes.ts`
- Logic: Houses the `Tracing Service`. Given a parameter like `cellId: "REV_2025_Q1"`, it queries the SQL relationships for `Transaction_Mappings` matching that aggregate string, returning 800 underlying micro-transactions arrays to the frontend.
4. `job.routes.ts`
- Logic: The master poller interface. Since calculations take 5-15 seconds, this route simply reads from `Prisma.Jobs.findUnique()` returning `{ status: 'running', progress: 45% }` to feed the React loader logic.

--------------------------------------------------------------------------------
PART 3: THE PYTHON-WORKER (THE BRAIN -> /python-worker)
--------------------------------------------------------------------------------

This is the mathematically rigorous FastAPI + Pandas application.

1. `hyperblock_engine.py`
- What it is: A multi-dimensional tensor framework substituting Excel logic.
- Mathematical Logic: Uses a `Sparse Matrix` concept across dictionaries. It tracks cells via `(Entity, Scenario, Time, Account)`. Instead of brute force summing, it calculates dependency paths. If `ARPU` changes, it only invalidates and recalculates nodes "downstream" (Revenue -> Gross Profit -> Net Income -> Retained Earnings), achieving O(n) performance rather than O(N^2) typical of spreadsheets.

2. `three_statement_engine.py`
- What it is: The strict GAAP accountant loop.
- Accounting Logic: Enforces `Assets = Liab + Equity`.
- Loop Pipeline: 
  A. Computes Revenue to Net Income for month M.
  B. Computes Net Working Capital (AR, AP, Inv) deltas.
  C. Calculates Operating Cash Flow.
  D. Adds CapEx -> Calculates Cash ending balance.
  E. Pushes ending cash and retained earnings mathematically down onto the Balance Sheet for month M. It throws a systemic `RuntimeError` if the balance sheet drifts by >$0.01.

3. `monte_carlo_enhanced.py`
- What it is: The stochastic probabilistic engine.
- Math Logic: Uses numpy vectorization `np.random.normal(loc=mu, scale=sigma, size=10000)`. It applies standard Brownian motion to variables like "Month-over-month growth" across 36 periods, calculating 10,000 parallel realities of the `hyperblock_engine.py`. It then runs `np.percentile([results], 10)` to yield the definitive 10th-percentile (P10) "worst-case" cash runway dates pushed to the frontend.

4. `reasoning_engine.py` / Agentic CFO
- What it is: Conversational LLM Orchestrator.
- Logic: Uses specialized agent roles. If queried on variance, it instantiates `SQLAnalyticsAgent`. It writes PostgreSQL statements on the fly against the Semantic Ledger (e.g., `SELECT * FROM ledger WHERE amount > 100 AND category='Cloud'`), retrieves the numeric anomaly, formats it within a reasoning chain, and returns it to `aicfo.routes.ts` containing `actionable=true` triggers.

5. Board Generating Analytics (`export_pptx.py` / `export_pdf.py`)
- What it is: Binary file output systems matching VC standards.
- Logic: Binds directly to `python-pptx`. It takes raw output from the Hyperblock array, determines charting scales using matplotlib, converts graphs to PNG buffers, and injects them onto pixel-perfect predetermined `Slide Layouts` alongside AI-drafted "Quarterly Sentiments." 

### SUMMARY OF END-TO-END FLOW:
1. User slides "Reduce Marketing 10%" in `realtime-simulations.tsx`.
2. React debounces heavily, sending a PATCH to the `backend`.
3. Backend queues a `Recalculate` job into PostgreSQL.
4. `python-worker` wakes up via `runner.py`, spinning up `hyperblock_engine.py`.
5. Hyperblock shifts $Marketing down, pushes through `three_statement_engine.py` to boost Net Income, retaining cash in Month 8.
6. The Python Engine completes, marking the job as "done" with the new JSON.
7. Next.js `pollModelRunStatus` catches the "done" state, pulls the JSON payload, feeding it directly into `Recharts`, showing the user a dynamically expanding cash runway graph. 

EVERY folder, logic chain, state hook, and mathematical process holds a highly specified role in resolving exactly this real-time loop.
